# ADR-0016: AI-анализ issues на z.ai GLM

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** как и чем автоматически анализировать issues (вероятная причина, приоритет, предложение по фиксу) в каждом проекте.

## Контекст

- Требование (2026-07-02): «нужен AI-агент, который анализирует issues в каждом проекте».
- По issue у нас уже богатый контекст: символикованный стек (ADR-0011), breadcrumbs, теги, окружение/релиз, сообщение исключения. LLM хорошо превращает это в короткий разбор: суть, вероятная причина, серьёзность, что чинить.
- Провайдер (решение пользователя): **z.ai (Zhipu), модель семейства GLM** — не Claude/Anthropic. z.ai даёт OpenAI-совместимый эндпоинт (`/api/paas/v4/chat/completions`, `Authorization: Bearer`). Elixir-проект → зовём по HTTP через `Req` (свои SDK/скиллы под Anthropic не используем).
- Инвариант: тяжёлое — не в HTTP-приёме и не в Broadway. LLM-вызов медленный и внешний → **фон** (Oban, ADR-0013), rate-limited, с ретраями.
- Ограничение ADR-0004: исходящий вызов, контейнер не добавляет. Фича опциональна: нет `ZAI_API_KEY` → анализ выключен, продукт работает как прежде.

## Решение

**Провайдер и вызов.** z.ai, OpenAI-совместимый chat-completions. Runtime-конфиг: `ZAI_API_KEY` (без него фича off), `ZAI_MODEL` (строка модели, дефолт `glm-4.6`; оператор переопределяет, напр. на `glm-5.2`), `ZAI_BASE_URL` (дефолт `https://api.z.ai/api/paas/v4`). Клиент `Swatter.AI.ZAI` — тонкая обёртка над `Req`: system+user сообщения, `response_format: {type: "json_object"}` для структурированного ответа, таймаут, разбор `choices[0].message.content`.

**Что анализируем и выход.** Промпт: роль «инженер-диагност»; вход — тип/сообщение исключения, символикованный in-app стек, breadcrumbs, окружение/релиз. Выход — строгий JSON: `summary` (1–2 фразы), `probable_cause`, `severity` (`low|medium|high|critical`), `suggested_fix`. Храним в отдельной таблице `issue_ai_analyses` (одна строка на issue, upsert): поля результата + `model`, `status` (`pending|ok|error`), `analyzed_at`. Отдельная таблица (не колонки в `issues`) — не раздуваем горячую таблицу, храним модель/версию, легко перезапускать (как releases в ADR-0011).

**Когда запускается (поправка 2026-07-03).** Только **по запросу**: `POST /api/0/issues/{issue_id}/analyze` (кнопка «Проанализировать» в деталке) ставит Oban-джобу `Swatter.AI.AnalyzeIssueWorker`; повторный вызов — повторный анализ. Авто-запуск на новый issue, изначально предполагавшийся, **отключён решением пользователя**: неконтролируемый расход токенов на шумном проекте (каждый новый fingerprint = вызов LLM). Поле `project_alert_settings.ai_enabled` зарезервировано под возможный будущий opt-in авто-режим и сейчас ничего не включает. Дедуп повторных кликов — уникальность Oban-джобы (окно 60 с).

**Воркер.** Очередь `ai` (отдельная от `alerts`, свой лимит конкуррентности — бережём rate limit z.ai). Берёт issue + последнее событие из ClickHouse (стек/breadcrumbs), строит промпт, зовёт `ZAI`, парсит JSON, upsert в `issue_ai_analyses`. Ошибка/невалидный JSON → `status: error`, лог, ограниченный ретрай Oban.

**Связь с алертами.** `NotifyWorker` (ADR-0013) при отправке подтягивает **ранее сделанный** анализ, если он есть, и вставляет `summary` в Telegram (best-effort; раз анализ по запросу — обычно алерт уходит без него). Алерт никогда не ждёт LLM.

**Тесты.** z.ai-HTTP мокается `Req.Test` (детерминированный JSON) — ключ для юнит/интеграционных тестов не нужен. Живой end-to-end против z.ai — при наличии `ZAI_API_KEY` (opt-in, как conformance).

## Альтернативы

- **Claude/Anthropic.** Отклонено решением пользователя в пользу z.ai GLM. Слой `Swatter.AI.*` провайдер-агностичен: сменить провайдера = поменять клиент+конфиг.
- **Анализ синхронно в пайплайне.** Отклонено: LLM-вызов — секунды, внешний, с лимитами; убил бы throughput и нарушил инвариант. Только фон.
- **Хранить анализ колонками в `issues`.** Отклонено: раздувает горячую таблицу, теряем модель/версию/повторы. Отдельная таблица — как releases (ADR-0011).
- **Свой/локальный LLM в контейнере.** Отклонено: нарушает «4 контейнера» (ADR-0004) и тяжёлый рантайм; внешний API — ноль инфраструктуры.
- **Официальный SDK.** В Elixir официального OpenAI/Anthropic SDK нет; `Req`-вызов к OpenAI-совместимому эндпоинту — минимально и в стеке.

## Последствия

- (+) Каждый issue получает разбор (суть/причина/severity/фикс) — быстрее триаж; `severity` помогает приоритизации и порогам алертов.
- (+) Провайдер-агностичный слой; z.ai сегодня, сменить — точечно.
- (+) Переиспользуем Oban (ADR-0013); фича опциональна, без ключа ядро не трогает.
- (−) Внешняя зависимость и секрет (`ZAI_API_KEY`), стоимость по токенам, задержка и возможные ошибки/лимиты провайдера — покрыто фоном/ретраями/`status`.
- (−) Данные ошибок (стек, сообщения) уходят во внешний сервис z.ai — приватность: явный opt-in ключом, документируется; на self-hosted решает оператор.
- Риск: формат/качество ответа модели — покрыто строгим JSON-форматом и `status: error` при невалидном ответе; продукт не блокирует.
