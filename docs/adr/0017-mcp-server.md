# ADR-0017: MCP-сервер для AI-агентов (удалённый дебаг по ссылке)

- **Статус:** Accepted
- **Дата:** 2026-07-03
- **Вопрос:** как дать Claude Code (и любому MCP-клиенту) доступ к данным Swatter, чтобы агент по ссылке на issue сам вытащил стектрейс/контекст и починил код локально.

## Контекст

- Сценарий: пользователь кидает агенту ссылку `https://хост/{org}/{proj}/issues/{id}` → агент получает символикованный стек, теги, breadcrumbs, AI-разбор, related errors и трейс → дебажит локальный репозиторий → закрывает issue. Sentry предлагает то же своим MCP — для нас это конкурентная поверхность поверх уже готовых данных (M1–M4).
- MCP Streamable HTTP — это JSON-RPC 2.0 поверх обычного HTTP POST; Claude Code подключает такие серверы: `claude mcp add --transport http … --header "Authorization: Bearer …"`.
- Ограничение ADR-0004: 4 контейнера. ADR-0007 давно запланировал API-токены `swt_*` — реализуются здесь.

## Решение

**Встроенный эндпоинт `POST /mcp`** в том же Phoenix-приложении (нулевая инфраструктура, переиспользуем контексты/авторизацию). Протокол реализуем сами (в духе ADR-0011 «свой декодер»): stateless-обработчик JSON-RPC — `initialize` (эхо поддерживаемой версии протокола), `notifications/*` → 202, `tools/list`, `tools/call`, `ping`; `GET /mcp` → 405 (SSE-стрим не предлагаем — спека это разрешает, ответы — обычный `application/json`). Батчи JSON-RPC не поддерживаем (убраны в MCP 2025-06-18).

**Авторизация — API-токены `swt_*`** (реализация отложенного из ADR-0007): 32 случайных байта, в БД только SHA-256 (та же таблица `user_tokens`, `context: "api"` + имя), плейнтекст показывается один раз при создании. Токен per-user: видимость MCP = организации пользователя (те же `member?`-проверки, что в dashboard API). Передача — `Authorization: Bearer swt_…`; невалидный токен → HTTP 401. Управление — страница API tokens в UI (создать/отозвать). Эти же токены — фундамент под sentry-cli (ADR-0012).

**Тулы MVP** (ответы — компактный markdown, оптимизированный под LLM):

| Тул | Вход | Отдаёт |
|---|---|---|
| `get_issue` | ссылка из UI или id | title/culprit/status/counts, символикованный стек с контекстом кода (верхний кадр первым, `*` = развёрнут из sourcemap), теги (env/release/host), breadcrumbs, AI-анализ (если делался), related errors трейса, ссылка на web |
| `list_issues` | project (slug), query?, status?, limit? | список issues с id — когда ссылки нет, только описание |
| `get_trace` | trace_id или ссылка | дерево спанов с длительностями (кросс-проектно) + ошибки трейса |
| `resolve_issue` | ссылка или id | помечает issue resolved — замыкает цикл «нашёл → починил → закрыл» |

Ошибки тулов (не найдено, нет доступа, кривой ввод) — `isError: true` с человекочитаемым текстом; чужие организации неотличимы от несуществующего («not found», как в dashboard API).

## Альтернативы

- **Отдельный npm-пакет-прокси (stdio).** Отклонено: дистрибуция, версионирование и лишний процесс у каждого пользователя; Claude Code умеет удалённый HTTP-транспорт нативно.
- **Библиотека MCP для Elixir (hermes-mcp и т.п.).** Отклонено для MVP: зависимость ради четырёх методов JSON-RPC; свой обработчик — пара сотен строк под полным контролем (паттерн ADR-0011).
- **OAuth-авторизация (как у hosted Sentry MCP).** Отклонено: тяжело для self-hosted MVP; Bearer-токен достаточен и проще в настройке.
- **SSE-стриминг ответов.** Отклонено: тулы отвечают быстро и атомарно; спека допускает plain-JSON ответы.

## Последствия

- (+) Swatter втыкается в Claude Code одной командой; агент дебажит по ссылке — символикация и AI-разбор уже сделаны сервером.
- (+) Появились API-токены — разблокирован sentry-cli (ADR-0012) и любая внешняя автоматизация.
- (−) Новая публичная поверхность — эволюция набора тулов и совместимость версий протокола на нас; смягчено минимализмом (4 тула, stateless).
- (−) Токен в конфиге агента — секрет у пользователя; ревокация есть, скоупы токенов (read-only) — возможное расширение.
