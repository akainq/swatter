# ADR-0002: Backend — Elixir (Phoenix + Broadway), а не Bun

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** открытый вопрос проекта «Bun или Elixir» для ingest-сервера и пайплайна обработки событий.

## Контекст

Профиль нагрузки error-monitoring-сервера:

- много одновременных мелких HTTP-запросов (события от SDK), пики при инцидентах у клиентов — именно в момент пика терять данные нельзя;
- пайплайн обработки: декодирование → нормализация → обогащение (sourcemaps) → группировка → батчевая запись в ClickHouse;
- долгоживущий stateful-процесс: буферы, батчеры, кэши ключей, счётчики rate limit;
- сбой обработки одного «ядовитого» события не должен ронять поток остальных.

Кандидаты: Bun (+Elysia) — один язык с фронтендом и JS SDK; Elixir — BEAM.

## Решение

**Elixir.** Ключевой аргумент: BEAM даёт из коробки ровно те свойства, которые для Bun пришлось бы строить руками — backpressure, супервизию, изоляцию сбоев, кластеризацию.

Состав:

- **Bandit/Plug (через Phoenix)** — HTTP ingest и dashboard API;
- **Broadway/GenStage** — пайплайн обработки с backpressure и батчингом в ClickHouse (канонический паттерн, проверенный Plausible Analytics на той же связке Elixir + ClickHouse);
- **Oban** — фоновые задачи: retention, дайджесты, доставка уведомлений с ретраями;
- **Ecto + PostgreSQL**, **`ch`/`ecto_ch`** (ClickHouse-клиент от команды Plausible), **Redix** (Redis);
- одно Phoenix-приложение с контекстами (`Ingest`, `Pipeline`, `Issues`, `Alerts`), не umbrella;
- scale-out: кластеризация BEAM (libcluster) — без изменения архитектуры.

Оговорка про CPU-bound: хэширование для группировки и разбор sourcemaps (JSON) — приемлемо на BEAM; разбор DWARF/minidump (нативная символикация C++/Rust) на BEAM не делаем — Rust sidecar или NIF, см. ADR-0011.

## Альтернативы

- **Bun + Elysia.** Плюсы: один язык (TS) на API, фронте и JS SDK; общие типы end-to-end; скорость старта; Elysia быстра. Минусы, решившие вопрос: event loop без изоляции — «ядовитое» событие или залипший обработчик влияет на весь процесс; backpressure и durability между приёмом и обработкой надо строить самим (внешний брокер обязателен с первого дня); воркеры примитивны, кластеризация — вручную; GC-паузы под постоянным потоком аллокаций. Для CRUD-API Bun отличен, но ядро продукта — именно пайплайн.
- **Гибрид: Elixir ingest + Bun dashboard-BFF.** Отклонено для старта: две runtime = двойная эксплуатация, деплой, найм. Вернёмся, если dashboard-у понадобится BFF (ADR-0008 фиксирует OpenAPI-контракт, так что миграция возможна).
- **Rust/Go ingest-edge + что-то для остального.** Преждевременная оптимизация; Sentry пришёл к Rust-edge (Relay) на масштабе SaaS, который не наш случай (self-hosted, ADR-0004).

## Последствия

- (+) Меньше движущихся частей: BEAM закрывает часть роли брокера/оркестратора; для self-hosted (ADR-0004) это критично.
- (+) Пики нагрузки: деградация управляемая — backpressure до 429 с `Retry-After`, официальные SDK умеют ретраи и локальную буферизацию.
- (−) Язык бэкенда ≠ язык фронтенда: контракт dashboard API фиксируем OpenAPI-спекой, типы для React генерируем из неё (ADR-0008).
- (−) Рынок Elixir-разработчиков уже; смягчение: кодовая база компактнее, паттерны стандартные (Phoenix/Broadway/Oban).
- (−) Нативная символикация уйдёт в отдельный Rust-компонент (ADR-0011) — примем это как осознанное исключение из «одна runtime».
- Примечание: Erlang и Elixir в списке целевых SDK-платформ — команда в любом случае работает с BEAM-экосистемой, порог входа ниже.
