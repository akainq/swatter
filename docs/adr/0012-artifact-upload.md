# ADR-0012: Загрузка и хранение sourcemap-артефактов

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** как клиент загружает sourcemap-артефакты и где сервер их хранит.

## Контекст

Для символикации (ADR-0011) сервер должен получить sourcemap до/независимо от событий и найти его по Debug ID. Развилка загрузки:

- **sentry-cli chunk-upload**: `sentry-cli sourcemaps upload` и bundler-плагины (`@sentry/vite-plugin` и др.) режут артефакты на SHA-1-чанки, шлют в `/chunk-upload/`, затем `/artifactbundle/assemble/`. Полная совместимость с CI/CD клиентов, но протокол тяжёлый (chunking, дедуп по SHA-1, сборка blob'а, поллинг статуса).
- **Свой простой endpoint**: multipart-загрузка бандла артефактов с их debug_id одним запросом.

## Решение

**Свой простой endpoint сейчас; chunk-upload — отдельным срезом поверх той же модели.**

- **Endpoint**: `POST /api/0/projects/{org}/{project}/artifacts` — multipart, поля: `file` (минифицированный js или .map), `debug_id`, `type` (`source_map` | `minified_source`), опц. `name`. Аутентификация — сессия/дальше API-токен (ADR-0007). Идемпотентно по `(project, debug_id, type)` — повторная загрузка заменяет.
- **Модель хранения**: PostgreSQL, таблица `artifact_bundles` (project_id, debug_id, type, name, размеры) + контент **gzip-сжатым `bytea`** в той же строке. Sourcemap хорошо жмётся (обычно сотни КБ–единицы МБ), в PG-бэкап попадает вместе со всем остальным — для self-hosted (ADR-0004, 4 контейнера) это не добавляет компонентов.
- **Лимит размера** артефакта в конфиге (по умолчанию 30 МБ распакованного); превышение → 413.
- **Debug ID извлекается клиентом**: bundler-плагин инжектит `debugId` в бандл и в поле `debugId` sourcemap; наш загрузчик/скрипт читает его и передаёт в форму. Документируем минимальный способ (vite-plugin + curl или наш скрипт).
- **Символикатор** (ADR-0011) читает контент по `(project, debug_id, type=source_map)`, распаковывает, кэширует.

## Альтернативы

- **Сразу chunk-upload** — полная совместимость с sentry-cli с первого дня, но сложный протокол откладывает ценность символикации; берём отдельным срезом (API-токены `swt_*` из ADR-0007 появятся там же).
- **Object storage (S3/MinIO)** для контента — 5-й компонент, нарушает ADR-0004; при масштабе (много крупных карт) вынесем за абстракцию хранилища (порог зафиксировать в BENCHMARKS). Для MVP PG bytea проще и бэкапится штатно.
- **Файловый volume у swatter-контейнера** — усложняет бэкап (отдельно от PG) и права; PG-путь единообразнее для одной ноды.

## Последствия

- (+) Символикация работает с первого среза M2, без тяжёлого протокола.
- (+) Хранилище не добавляет контейнеров и попадает в общий PG-бэкап.
- (+) Та же таблица `artifact_bundles` — целевая и для chunk-upload среза: меняется только транспорт загрузки, не модель и не символикатор.
- (−) sentry-cli/bundler-плагины со стандартным upload не заработают до chunk-upload среза — нужен наш скрипт/адаптер; документируем.
- (−) Крупные карты в PG раздувают БД; лимит размера + оговорка про object storage при масштабе.
