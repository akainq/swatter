# ADR-0009: Rate limiting приёма — fixed window в Redis, per-key

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** как ограничивать поток событий на приёме, не теряя их молча и оставаясь совместимыми с поведением Sentry SDK.

## Контекст

Инцидент у клиента — это шторм одинаковых событий; без лимитов один проект может утопить инсталляцию (ADR-0004: один узел). Протокол Sentry задаёт форму отказа: `429` + `Retry-After` (+ опционально `X-Sentry-Rate-Limits: <retry_after>:<categories>:<scope>`), и официальные SDK на неё правильно реагируют — буферизуют, ретраят, считают потери в client reports. Значит «отказ по лимиту» — это управляемая деградация, а не потеря.

## Решение

- **Гранулярность v1 — DSN-ключ** (`project_keys`): лимитируется приём envelope целиком (не по категориям items). Категорийные лимиты (errors отдельно от transactions) — при появлении квот (пост-MVP).
- **Алгоритм — fixed window в Redis**: `INCR swatter:rl:{key_id}:{window_start}` + `EXPIRE`. Одна пайплайн-команда на запрос, Redis уже в горячем пути (ADR-0005).
- **Лимиты**: дефолт в конфиге (`3000 событий / 60 с` на ключ), переопределение per-key колонками `rate_limit_count` / `rate_limit_window_seconds` (NULL = дефолт). UI управления — вместе с экраном настроек проекта.
- **Ответ при превышении**: `429`, `Retry-After: <до конца окна>`, `X-Sentry-Rate-Limits: "<retry_after>::key"` (пустые категории = все).
- **Fail-open**: при недоступном Redis лимитер пропускает. Это осознанно: если Redis лежит, следующий шаг (буфер, ADR-0005) всё равно ответит 503 — двойной отказ не нужен, а ложные 429 при сетевой икоте вредны.
- Проверка стоит **после** аутентификации ключа и **до** чтения тела — превышенный лимит не тратит полосу на upload.

## Альтернативы

- **Sliding window / token bucket** — точнее на границах окон и плавнее, но дороже (ZSET/Lua) и сложнее объяснять пользователю. Граничный всплеск ×2 в v1 принимаем.
- **Готовая библиотека (Hammer и т.п.)** — внутри тот же fixed window; лишняя зависимость при готовом Redix.
- **Лимиты в Postgres** — тащит горячий путь в control-plane БД. Отклонено.
- **Лимит по IP** — бессмысленно для серверных SDK за NAT; ключ — правильная граница ответственности.

## Последствия

- (+) Шторм срезается предсказуемо и без молчаливых потерь (SDK знают про 429).
- (+) Форма ответа совместима с официальными SDK и масштабируется на будущие категорийные лимиты.
- (−) Fixed window допускает до 2× лимита на границе окон — принято для v1.
- (−) Счётчик общий на ключ, не на инстанс SDK — «шумный сосед» внутри одного проекта не изолируется (это и не цель).
