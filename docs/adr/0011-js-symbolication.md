# ADR-0011: JS-символикация — Debug IDs, sourcemap на BEAM

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** как разворачивать минифицированный JS-стектрейс в исходные позиции.

## Контекст

В production JS исполняется минифицированным: стектрейс события несёт бесполезные `app.min.js:1:24578`. Чтобы показать исходный файл/строку/функцию, нужен sourcemap и способ связать минифицированный фрейм с картой.

Два механизма связывания:

1. **Release + URL** (legacy): артефакт привязан к релизу по URL (`~/static/js/app.min.js`), фрейм матчится по `release` события + `abs_path`. Хрупко: зависит от точного `release` в событии и от путей/CDN, ломается при хешированных именах.
2. **Debug IDs** (современный Sentry, 2023+): и бандл, и sourcemap помечаются одним уникальным Debug ID (UUID), инжектится bundler-плагином/sentry-cli на сборке. Событие несёт `debug_meta.images[].debug_id` (или `//# debugId=` в самом бандле → SDK извлекает). Матчинг по Debug ID, а не по URL.

## Решение

**Debug IDs как единственный механизм в MVP.** Матчинг фрейма к sourcemap — по Debug ID из `debug_meta` события, не по release/URL (ADR-0006 fingerprint и M2-загрузка тоже строятся вокруг debug_id).

- **Парсинг sourcemap — на BEAM, свой декодер** Source Map v3 (Base64 VLQ mappings): бинарный поиск по generated-позиции фрейма → source/line/name. ~150 строк, без зависимостей; sourcemap это JSON, разбор дешёв (оговорка про CPU в ADR-0002 касалась нативных DWARF, не JS).
- **Кэш разобранных карт** в ETS: карта разбирается один раз на debug_id, дальше — переиспользуется (шторм одинаковых событий не переразбирает мегабайтную карту).
- **Символикация в пайплайне** (Broadway, ADR-0002): между нормализацией и fingerprint. Если карты для debug_id нет — фрейм остаётся минифицированным (не ошибка), символикация «best effort».
- **Fingerprint после символикации:** развёрнутые in-app фреймы дают стабильную группировку между сборками (минифицированные имена меняются от билда к билду). Grouping version инкрементируется (ADR-0006), когда символикация включается для проекта.
- Каждый развёрнутый фрейм помечается `in_app`/`data.symbolicated` для UI (ADR: отметка на фрейме).

## Альтернативы

- **Release + URL** — отклонён как основной: хрупок, требует дисциплины release/путей; современные SDK и так инжектят debug_id. Возможен как fallback позже (отдельный ADR), не в MVP.
- **Готовый hex-пакет для sourcemap** — экосистема бедна и заброшена; свой декодер компактнее и контролируем.
- **sentry/symbolicator (Rust)** — рассчитан на нативные (DWARF/minidump), не на JS; тянет Rust-сервис и FSL-лицензию (см. ADR-0001). Для JS избыточен.
- **Нативная символикация (C++/Rust minidump, DWARF)** — по-прежнему **пост-MVP**: не на BEAM, Rust sidecar/NIF, отдельный ADR при взятии в работу.

## Последствия

- (+) Матчинг не зависит от путей развёртывания/CDN — работает «из коробки» с современными bundler-плагинами.
- (+) Ноль новых компонентов и зависимостей: разбор и кэш на BEAM.
- (−) Старые SDK/сборки без debug_id не символикуются в MVP (нужен fallback — позже).
- (−) Свой VLQ-декодер — наш код на поддержке; смягчение: узкая, хорошо специфицированная задача с юнит-тестами на реальных картах.
- (−) Символикация в горячем пайплайне: при промахах кэша разбор большой карты стоит CPU — ограничиваем размер карты и кэшируем агрессивно.
