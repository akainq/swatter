# ADR-0007: Аутентификация и авторизация dashboard

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** модель пользователей и доступов для UI и dashboard-API. (Аутентификация ingest — DSN-ключи — здесь ни при чём: ADR-0001/0009.)

## Контекст

Self-hosted инсталляция (ADR-0004): установка должна доводить до рабочего состояния без внешнего IdP и SMTP. Модель данных мультитенантна с первого дня (`org_id` везде), продукт в MVP — single-tenant.

## Решение

- **Сущности**: `users`, `organizations`, `memberships` (роль `owner | admin | member`), проекты принадлежат организации. Teams — post-MVP.
- **Первый запуск**: если `users` пуста — UI предлагает создать первого пользователя (owner) и организацию. Без подтверждения email (SMTP может отсутствовать).
- **Сессии, не JWT**: пароли — bcrypt (`bcrypt_elixir`, схема в духе `phx.gen.auth`); сессия — HttpOnly cookie. SPA отдаётся тем же Phoenix-приложением → same-origin: ни CORS, ни token-refresh-машинерии.
- **API-токены** (для CI/скриптов/sentry-cli в M2): персональные и организационные, хранятся как хэш, показываются один раз, имеют префикс вида `swt_` для идентификации в логах/сканерах секретов; скоупы — на чтение/запись по org.
- **Авторизация**: роль на уровне организации; проверка — плагом на границе API (`current_membership` в conn). Пер-проектные права — post-MVP.
- **SSO (OIDC), 2FA, SCIM** — post-MVP (см. ROADMAP), схема сущностей им не противоречит.

## Альтернативы

- **JWT/stateless** — выгоды при same-origin SPA нет, а ревокация и инвалидация сессий усложняются. Отклонено.
- **Сразу OIDC-only** — ломает «поставил и работает» self-hosted без IdP. Отклонено для MVP.
- **Внешний IdP в комплекте (Keycloak/Zitadel в compose)** — пятый контейнер, нарушает ADR-0004. Отклонено.

## Последствия

- (+) Первый запуск без внешних зависимостей; минимум движущихся частей.
- (+) Same-origin упрощает и SSE (ADR-0008): куки работают из коробки.
- (−) Восстановление пароля без SMTP невозможно — компенсация: CLI-команда сброса пароля (`mix swatter.reset_password`) для админа сервера.
- (−) API-токены требуют аккуратного аудита (последнее использование, отзыв) — закладывается в схему сразу.

## Поправки

- **2026-07-02, pbkdf2 вместо bcrypt.** `bcrypt_elixir` — C-NIF и требует
  MSVC-toolchain на Windows-машинах разработки. Используем `pbkdf2_elixir`
  (чистый Elixir поверх `:crypto`, PBKDF2-SHA512, интерфейс Comeonin тот
  же). Замена на bcrypt/argon2 при необходимости — одна строка в
  `User.hash_password/1` плюс перехэширование при следующем логине.
- **2026-07-02, реализация в срезе 4.** Сессии реализованы токенами в БД
  (`user_tokens`, в базе — SHA-256 от токена, в cookie — сырой) — ревокация
  при logout/сбросе пароля работает. CSRF закрыт `SameSite=Lax` + JSON-only
  API. API-токены (`swt_*`) перенесены в M2 — появятся вместе с
  sentry-cli-совместимостью (ADR-0012), раньше они не нужны.
