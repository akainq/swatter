# ADR-0006: Группировка событий в issues (fingerprint v1)

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** по какому правилу отдельные события агрегируются в issues.

## Контекст

Группировка — сердце продукта: слишком грубая склеивает разные баги в один issue, слишком тонкая дробит один баг на десятки issues (и спамит алертами). Требования: детерминированность (одно событие всегда попадает в одну группу), стабильность между деплоями (номера строк меняются от релиза к релизу — в хэш им нельзя), уважение к явному `fingerprint` из SDK (часть протокола Sentry), понятность поведения для пользователя.

Ограничение v1: перегруппировка задним числом невозможна (события в ClickHouse иммутабельны) — смена алгоритма действует только на новые события.

## Решение

Приоритет источников fingerprint (первый применимый):

1. **Явный `fingerprint`** из события; плейсхолдер `{{ default }}` заменяется на результат правил ниже (совместимо с семантикой Sentry).
2. **Стектрейс исключения**: для всей цепочки chained exceptions — тип исключения (без значения!) + последовательность in-app фреймов в виде пар `(module, function)`. Нормализация: номера строк/колонок отбрасываются; anonymous-функции BEAM (`-fun.3-` и т.п.), лямбды и index-суффиксы срезаются; адреса/hex-литералы в именах вычищаются. Если in-app фреймов нет — берутся все фреймы.
3. **Message-шаблон**: если стектрейса нет — `logentry.message` (шаблон до интерполяции параметров), иначе сырое `message` с вычисткой чисел/UUID/hex.
4. **Fallback**: `level` + первые 255 символов нормализованного message.

Из выбранных компонент считается SHA-256 → `issues.fingerprint_hash`; уникальность — `(project_id, fingerprint_hash)`. Метаданные группы: `title` = тип исключения + усечённое значение, `culprit` = верхний in-app фрейм.

Версионирование: `issues.grouping_version` (int). Изменение алгоритма = инкремент версии; старые issues продолжают жить, новые события группируются новой версией (поведение Sentry). Никакого тихого «умного» слияния в v1.

## Альтернативы

- **Хэш с номерами строк** — каждый деплой порождает новые issues. Отклонено.
- **Только message** — склеивает разные баги с одинаковым текстом и дробит по интерполированным значениям. Отклонено как основной путь (остаётся fallback-ступенью).
- **Значение исключения в хэше** — `ValueError: id 123 not found` и `id 456` стали бы разными issues. Отклонено.
- **ML/embedding-похожесть** — недетерминированно, необъяснимо для пользователя. Возможна позже как *вторичная* фича «похожие issues», не как группировка.

## Последствия

- (+) Детерминированно, объяснимо, воспроизводимо в тестах; совместимо с явными fingerprint из Sentry SDK.
- (−) До символикации (M2) минифицированные JS-стеки группируются по минифицированным именам — грубо; после включения символикации fingerprint считается по развёрнутому стеку, и версия группировки инкрементируется.
- (−) Смена алгоритма дробит непрерывность issues — смягчается `grouping_version` и редкостью изменений.
- Тестовый корпус: фикстуры реальных событий от всех целевых SDK (из conformance-матрицы M1) с ожидаемыми fingerprint — регресс-тесты алгоритма.
