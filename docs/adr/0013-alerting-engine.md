# ADR-0013: Движок алертов — правила, дедуп, Telegram-доставка

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** как уведомлять команду о важных событиях (новый issue, регрессия, всплеск частоты) и через какой канал.

## Контекст

- M3 (ROADMAP): «команда узнаёт о новой ошибке из Telegram раньше, чем из UI».
- Продуктовое решение (2026-07-02): единственный канал — **Telegram** (email/Slack/webhook в MVP не делаем). Причина: self-hosted, минимум внешней инфраструктуры и настройки. Telegram Bot API — простой HTTP, без SMTP/OAuth, мгновенная доставка, бесплатно.
- Инвариант дизайна: приём не делает тяжёлой работы в HTTP; асинхронная обработка — в пайплайне/фоне. Алерт — внешний HTTP-вызов с ретраями → это **фон**, не Broadway (пайплайн throughput-критичен).
- ADR-0002 предусматривает Oban для фоновых задач — здесь он и появляется.
- Ограничение ADR-0004 (4 контейнера): Telegram — исходящий вызов на `api.telegram.org`, контейнер не добавляет; Oban живёт в PostgreSQL (уже есть).

## Решение

**Канал.** Только Telegram. Бот — общий на инстанс: `TELEGRAM_BOT_TOKEN` в runtime-конфиге. Маршрутизация — per-project `chat_id`: каждый проект шлёт в свой чат/канал/группу. Нет токена или нет `chat_id` у проекта → алерты проекта выключены (fail-safe; фича опциональна).

**Правила** — per-project настройки, таблица `project_alert_settings` (одна строка на проект):

- `on_new_issue` — первое появление fingerprint (issue создан).
- `on_regression` — resolved-issue вернулся в более новом релизе (ADR-0011, флаг `regressed`).
- `frequency_threshold` + `frequency_window_seconds` — N событий по одному issue за окно T (всплеск).

Все три включаются независимо. Дефолт при создании проекта: `on_new_issue=true`, `on_regression=true`, порог выключен.

**Триггер.** `Issues.upsert_from_event/4` возвращает вид события: `:new | :regression | :ongoing`. Пайплайн (`persist_event`) на `:new`/`:regression`, если у проекта включено соответствующее правило, ставит Oban-джобу `Swatter.Alerts.NotifyWorker`. Порог частоты — счётчик в Redis per-issue-per-window; при пересечении порога джоба ставится там же. Постановка джобы — быстрая вставка в PG: инвариант «пайплайн не делает тяжёлой работы» сохранён (сам HTTP к Telegram — внутри воркера).

**Дедуп/cooldown.** Чтобы шумный issue не залил чат: per-(issue, rule) cooldown в Redis (`SET NX EX`, тот же паттерн, что rate limiting ADR-0009). Пока ключ жив — повторный алерт того же типа по тому же issue не ставится. Дефолт cooldown — 15 минут, конфигурируемо. `on_new_issue` по природе однократен, но cooldown защищает от гонок.

**Доставка (Oban).** Очередь `alerts`. Воркер собирает контекст (issue, проект, релиз, ссылка на деталку, при наличии — AI-резюме из ADR-0016), форматирует сообщение (Markdown) и делает `POST https://api.telegram.org/bot<token>/sendMessage` через `Req`. Ретраи Oban на сеть/5xx Telegram; 4xx (кривой `chat_id`) не ретраим, лог. Требование «терять нельзя» покрывается ретраями Oban.

**Тест-режим.** Oban в тестах — `testing: :manual` (джобы не исполняются, проверяем факт постановки); Telegram-HTTP мокается `Req.Test`. Пайплайн в test не стартует (как и раньше), интеграционные тесты поднимают его в sandbox и проверяют enqueue.

## Альтернативы

- **Email/Slack/webhook (как в Sentry).** Отклонено продуктовым решением: для self-hosted MVP Telegram закрывает «мгновенно узнать» без SMTP и настройки incoming webhooks. Расширение каналов — пост-MVP; `NotifyWorker` абстрагирует канал, точка расширения заложена.
- **Доставка прямо из пайплайна (без Oban).** Отклонено: внешний HTTP в Broadway блокирует обработку и роняет throughput; ретраи пришлось бы городить руками. Oban предусмотрен ADR-0002.
- **Cooldown/дедуп в PG.** Отклонено: PG-запись на каждое событие дорого; Redis `SET NX EX` — тот же паттерн, что ADR-0009. Настройки правил (меняются редко) — в PG.
- **Per-project bot-токен.** Отклонено для MVP: общий бот проще в эксплуатации (один токен на инстанс), per-project — только `chat_id`. Per-project токен — возможное расширение.

## Последствия

- (+) Уведомления в реальном времени с минимумом настройки: создать бота, вписать токен, указать `chat_id` проекта.
- (+) Появляется Oban — переиспользуется для AI-анализа (ADR-0016) и будущих фоновых задач.
- (+) Точка расширения каналов заложена, не ломая MVP.
- (−) Новая внешняя зависимость (`api.telegram.org`) и секрет (bot token) в эксплуатации; при недоступности Telegram алерты копятся в ретраях Oban.
- (−) Ещё одна таблица настроек и Oban-таблицы в PG-миграциях.
- Риск: пороговые правила на шумных проектах могут спамить — покрыто cooldown; тонкая настройка порогов — на операторе.
