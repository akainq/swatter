# ADR-0001: Совместимость с Sentry на уровне ingest-протокола

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** как обеспечить «простую интеграцию» и обратную совместимость с Sentry для платформ Node.js/Bun, Go, Python, C++, Rust, Erlang, Elixir.

## Контекст

Цель продукта — чтобы существующее приложение, уже инструментированное Sentry, переехало на Swatter с минимальными изменениями. Экосистема Sentry SDK зрелая и покрывает все целевые платформы:

| Платформа | SDK |
|---|---|
| Node.js | `@sentry/node` (официальный) |
| Bun | `@sentry/bun` (официальный) |
| Go | `sentry-go` (официальный) |
| Python | `sentry-python` (официальный) |
| C++ | `sentry-native` (официальный; включая minidumps) |
| Rust | crate `sentry` (официальный) |
| Elixir | `sentry` hex-пакет (официальный sentry-elixir) |
| Erlang | официального нет; community-клиенты — проверить в M1, при необходимости тонкая обёртка над HTTP-протоколом |

Писать и поддерживать собственные SDK под 7+ платформ — постоянная стоимость, сравнимая с самим сервером.

## Решение

Реализуем **серверную сторону протокола приёма Sentry** (`sentry_version=7`). Интеграция = замена DSN в конфиге SDK.

Поверхность совместимости (MVP):

- **DSN**: `{protocol}://{public_key}@{host}/{project_id}`.
- **Endpoints**:
  - `POST /api/{project_id}/envelope/` — основной (envelope-формат: JSON-заголовок + items);
  - `POST /api/{project_id}/store/` — legacy JSON-события (старые SDK);
  - `POST /api/{project_id}/minidump/` — фаза нативной символикации (пост-MVP);
  - `POST /api/{project_id}/security/` — CSP-отчёты (опционально, пост-MVP).
- **Аутентификация**: заголовок `X-Sentry-Auth` (`sentry_key=...`) и query-параметр `?sentry_key=`; секретный ключ не требуем (deprecated в Sentry).
- **Кодирование**: gzip тела обязателен к поддержке.
- **Envelope items в MVP**: `event`, `transaction`, `session`/`sessions`, `client_report`, `attachment`. **Неизвестные типы items принимаются и молча отбрасываются** — envelope не отклоняется (правило forward compatibility протокола).
- **Ответы**: `200 {"id": "<event_id>"}`; `429` + `Retry-After`; заголовок `X-Sentry-Rate-Limits: <seconds>:<categories>:<scope>` для покатегорийных лимитов.
- **Часы**: учитываем `sent_at` из заголовка envelope для коррекции дрейфа часов клиента.
- **Tunnel-режим** SDK (проксирование envelope через бэкенд приложения) работает автоматически — на сервер приходит тот же envelope.

Спецификация протокола: https://develop.sentry.dev/sdk/ — отслеживаем изменения.

Обязательство качества: **conformance-тесты против официальных SDK** — матрица мини-приложений на каждой целевой платформе шлёт типовые события в Swatter, тест проверяет результат через API (см. ROADMAP, M1).

## Альтернативы

- **Собственные SDK с Sentry-подобным API** — drop-in замена пакетов. Отклонено: 7+ SDK на поддержке навсегда; ценность против замены DSN мала.
- **OTLP-first** (OpenTelemetry как единственный вход). Отклонено как основной путь: не даёт drop-in совместимости с Sentry SDK и слабее для crash-репортинга (minidumps, sessions). OTLP-ingest возможен позже как второй вход (отдельный ADR).
- **Переиспользовать/форкнуть Sentry Relay** как edge. Отклонено: лицензия FSL (запрещает конкурирующее использование), Rust-компонент вне нашего стека, привязка к внутренностям Sentry.

## Последствия

- (+) Все целевые платформы работают с первого дня; ноль собственного SDK-кода.
- (+) Миграция с Sentry — одна строка конфига; обратная миграция тоже (нет lock-in — доверие пользователей).
- (−) Протокол эволюционирует на стороне Sentry — нужен процесс отслеживания и conformance-CI.
- (−) Мы не контролируем клиентское поведение SDK (буферизация, повторы, сэмплинг на клиенте) — сервер должен корректно обрабатывать всё, что шлют реальные SDK, включая старые версии.
- (−) Функциональность, требующая изменений SDK (свои поля, свои фичи), недоступна — только серверные расширения.
