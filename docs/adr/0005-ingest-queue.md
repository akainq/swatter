# ADR-0005: Буферизация между приёмом и обработкой событий

- **Статус:** Accepted (спайк 2026-07-02 подтвердил вариант B)
- **Дата:** 2026-07-02
- **Вопрос:** нужен ли durable-буфер между HTTP-приёмом envelope и пайплайном обработки, и какой.

## Контекст

Принцип ingest: ответить SDK `200` как можно быстрее, всю работу делать асинхронно. Между «приняли» и «записали в ClickHouse» событие живёт в буфере. Вопросы: переживает ли буфер рестарт/крэш процесса; что происходит при пике (backpressure); сколько компонентов это добавляет в self-hosted-инсталляцию (ограничение ADR-0004: 4 контейнера).

## Варианты

**A. Без внешнего брокера: Broadway-буфер в памяти BEAM.**
Приём кладёт событие напрямую в GenStage-пайплайн. Просто, ноль новых компонентов, максимальная скорость.
− При крэше/рестарте процесса теряются события в полёте (окно — секунды, но при деплое — гарантированно). Можно добавить свой WAL на диск, но это «написать маленький брокер самим».

**B. Redis Streams (рекомендация).**
Redis уже в стеке (ADR-0003). Приём: auth → лимиты → `XADD` → `200`. Пайплайн: Broadway-консьюмер с consumer group (community-адаптер `off_broadway_redis_stream` либо свой GenStage-producer, ~200 строк), ack после записи в CH.
+ Durability между приёмом и обработкой (AOF), рестарты и деплой без потерь, ноль новых контейнеров, ёмкость (десятки тысяч msg/s) с запасом покрывает потолок одной ноды из ADR-0004.
− Redis становится stateful-критичным (нужен AOF и лимит длины стрима с политикой сброса в 429).

**C. Kafka / Redpanda.**
Максимальные гарантии и масштаб, replay истории.
− Пятый контейнер, JVM-эксплуатация (Kafka) или новая зависимость (Redpanda); для потолка нагрузки self-hosted — оверкилл. Путь миграции при росте: сменить транспорт за Broadway-абстракцией, отдельным ADR.

**D. NATS JetStream.**
Лёгкий (один бинарь), durable.
− Всё равно +1 компонент и +1 технология в эксплуатации ради того, что Redis уже даёт на нашем масштабе.

## Решение

**B (Redis Streams)**: единственный вариант одновременно durable и не добавляющий компонентов. Broadway абстрагирует транспорт — переход на C при реальной потребности не меняет пайплайн. Консьюмер — `off_broadway_redis_stream` (0.10.0, Apache-2.0, развивается с 2022); fallback, если упрёмся в ограничения адаптера, — собственный GenStage-producer (~200 строк).

### Спайк (2026-07-02, Redis 8 в Docker, dev-машина, замер изнутри контейнера)

| Операция | Результат |
|---|---|
| `XADD` 200B payload, 50 соединений, без pipeline | ~153 000 ops/s, p50 0.17 мс |
| `XADD` 1KB payload, pipeline 16 | ~270 000 ops/s |
| `XREADGROUP COUNT 10` (consumer group) | ~33 700 вызовов/с ≈ 330 000 событий/с доставки |
| Память: 200 000 записей (~120 MB payload) | ~135 MB — оверхед стрима незначителен |

Запас относительно потолка одной ноды из ADR-0004 (единицы тысяч событий/с) — **два порядка**; буфер не станет узким местом раньше обработки и ClickHouse. AOF включён в `compose.yaml` с первого дня.

## Последствия

- (+) Деплой без потери событий; изоляция приёма от обработки.
- (−) Конфигурация Redis обязана включать AOF; максимальная длина стрима и поведение при переполнении (backpressure → 429) фиксируются в конфиге по умолчанию.
