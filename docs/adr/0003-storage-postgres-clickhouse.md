# ADR-0003: Хранилища — PostgreSQL (control plane) + ClickHouse (события) + Redis

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** где хранить события (append-only, большие объёмы, аналитика) и метаданные (транзакционные, мутабельные).

## Контекст

Два принципиально разных профиля данных:

1. **События** (errors, transactions, sessions): append-only, высокая кардинальность, миллионы строк, запросы аналитические — top issues за период, счётчики по тегам, перцентили длительностей, поиск по атрибутам. Мутации почти не нужны, нужен дешёвый TTL.
2. **Control plane**: организации, проекты, DSN-ключи, пользователи, issues (как сущности со статусом), релизы, правила алертов. Мало строк, транзакционность, частые точечные обновления.

## Решение

- **PostgreSQL** — control plane: `organizations`, `projects`, `project_keys`, `users`/`memberships`, `issues` (fingerprint, статус, assignee, денормализованные `first_seen`/`last_seen`/`times_seen`), `releases`, `artifact_bundles`, `alert_rules`, `notification_channels`, `api_tokens`.
- **ClickHouse** — события: `events`, `transactions`/`spans`, `sessions` (агрегаты для crash-free rate). MergeTree, `ORDER BY (project_id, issue_id, timestamp)` для events, TTL по retention-политике (ADR-0010). Батчевые вставки только из пайплайна (Broadway), никаких вставок по одной строке.
- **Redis** — rate limiting, кэш DSN-ключей, дедупликация/cooldown алертов, кандидат на роль ingest-буфера (ADR-0005).
- Связь между хранилищами: JOIN между PG и CH нет — измерения, нужные в аналитике (release, environment, issue_id), денормализуются в строки CH при записи. Счётчики issues в PG обновляются пайплайном батчами (не на каждое событие).

## Альтернативы

- **Только PostgreSQL** (партиционирование, BRIN). Отклонено: аналитические запросы по десяткам миллионов событий с фильтрами по тегам — не профиль PG; это главная причина, почему ранний Sentry (который так и работал) мигрировал на ClickHouse.
- **Только ClickHouse**. Отклонено: control plane требует транзакций и мутаций (статусы issues, membership), mutations в CH дороги и асинхронны.
- **Elasticsearch** вместо CH. Отклонено: дороже в эксплуатации (JVM, память), слабее в агрегациях/перцентилях, лицензионные нюансы.
- **TimescaleDB** («остаться в одном PG»). Рассмотрено всерьёз ради простоты self-hosted; отклонено: компрессия и скорость агрегаций CH на этом профиле данных на порядок лучше, а CH в docker-compose — один контейнер без ZooKeeper (реплика не нужна в MVP).

## Последствия

- (+) Проверенная связка: Sentry (Snuba/ClickHouse + PG), PostHog, Plausible — тот же паттерн; ошибки первопроходцев уже описаны.
- (+) Retention — бесплатный (TTL в CH), счётчики и графики быстрые.
- (−) Три хранилища в docker-compose (PG, CH, Redis) — цена принята в ADR-0004; официальный self-hosted Sentry для сравнения требует ещё Kafka+ZooKeeper и ~16 контейнеров.
- (−) Дублирование измерений в CH: изменение имени релиза/окружения задним числом не отражается в исторических строках — принимаем (у Sentry так же).
- (−) Нужна дисциплина батчинга вставок в CH — закреплена архитектурно в пайплайне (ADR-0002, ADR-0005).
