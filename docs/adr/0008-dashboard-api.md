# ADR-0008: Dashboard API — REST в стиле Sentry + OpenAPI + SSE

- **Статус:** Accepted
- **Дата:** 2026-07-02
- **Вопрос:** контракт между Phoenix и React SPA (и будущими внешними потребителями API).

## Контекст

Язык бэкенда ≠ язык фронтенда (ADR-0002) — типы не шарятся компилятором, контракт нужен явный. Вторая сила: у Sentry есть устоявшаяся форма web-API (`/api/0/...`, cursor-пагинация через `Link`-заголовок), на которую завязаны существующие интеграции и tooling; близость к этой форме удешевляет будущую совместимость (sentry-cli в M2 — ADR-0012, сторонние дашборды — post-MVP).

## Решение

- **REST JSON под префиксом `/api/0/`** — пути и формы в стиле Sentry там, где сущности совпадают: `/api/0/organizations/{org}/...`, `/api/0/projects/{org}/{project}/issues/...`. Полную совместимость web-API в MVP **не обещаем** — обещаем узнаваемость и отсутствие конфликта путей с ingest (`/api/{project_id}/envelope/` различается числовым сегментом; роутер держит их в разных scope).
- **Пагинация** — cursor-based через `Link`-заголовок (форма Sentry). Никаких offset.
- **Ошибки** — `{"detail": "..."}` (форма Sentry) + корректные статус-коды.
- **Контракт — OpenAPI** (`open_api_spex`): спека генерируется из кода, публикуется по `/api/0/openapi.json`; TS-типы и клиент для `web/` генерируются из неё (`openapi-typescript`) на CI — расхождение контракта ловится сборкой.
- **Live-обновления — SSE** (`text/event-stream`, chunked из Phoenix): поток новых событий/issues для открытых списков и деталки. Однонаправленности достаточно; работает через любые прокси/LB, куки same-origin (ADR-0007) — из коробки.
- Версионирование: `/api/0/` считается нестабильным до 1.0; после — additive-изменения без слома.

## Альтернативы

- **GraphQL (absinthe)** — гибкость запросов не нужна (формы экранов известны), а сложность квот/кэширования/N+1 — реальная. Отклонено.
- **Phoenix Channels/WebSocket** — двунаправленность не нужна; тянет клиентскую библиотеку и усложняет прокси. Отклонено для live-обновлений; не исключено для будущих интерактивных фич.
- **Свободный REST без оглядки на Sentry** — теряем дешёвый путь к совместимости tooling при нулевой экономии сейчас. Отклонено.
- **tRPC-стиль** — привязывает контракт к TS-серверу, которого нет. Отклонено.

## Последствия

- (+) Типобезопасный фронт без ручной синхронизации типов; контракт проверяется CI.
- (+) Форма Sentry снижает стоимость ADR-0012 (sentry-cli) и сторонних интеграций.
- (−) Дисциплина: каждый endpoint описывается в open_api_spex (иначе теряется главный бонус).
- (−) SSE-соединения — долгоживущие процессы BEAM: дёшево, но требует лимита на пользователя и heartbeat.
