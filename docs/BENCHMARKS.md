# Замеры производительности

Требование вехи M1 — зафиксировать потолок одной ноды (ADR-0004).

## 2026-07-02 — первичный замер ingest (Windows dev-машина)

**Стенд:** Intel Core Ultra 9 275HX (24 потока), 64 GB RAM, Windows 11;
PG/CH/Redis — Docker Desktop (WSL2); сервер — `MIX_ENV=prod mix phx.server`
на хосте. Клиент — bombardier 2.0.2, envelope 861 B (event-item 700 B),
`POST /api/1/envelope`, rate limit ключа отключён.

| Замер | Результат |
|---|---|
| c=16, 2 000 запросов | 260 rps, latency avg 62 мс, 100% `200` |
| c=128, 10 000 запросов | **715 rps**, latency avg 184 мс, 100% `200`, 0 ошибок |
| Контроль: `OPTIONS` (ни PG, ни Redis) | latency avg **54 мс** |
| Пайплайн (12 000 событий) | обработаны в issue **без накопления backlog** — дренаж ≥ ingest |
| Redis Streams (спайк ADR-0005) | XADD ~150k ops/s (без pipeline), ~270k ops/s (pipeline 16) |

### Интерпретация

Замер **ограничен клиентской стороной, а не сервером**: no-op `OPTIONS`
без единого обращения к базам показывает те же ~54 мс фиксированной
задержки — это артефакт Windows loopback + Docker Desktop (delayed
ACK/NAT WSL2), он доминирует в каждом запросе.

## 2026-07-02 — ingest внутри Linux/docker-сети (деплой-срез)

**Стенд:** та же машина, но полный релизный стек
(`docker-compose.prod.yaml`, релиз prod); клиент — `alpine/bombardier`
в том же docker-network, запрос `http://swatter:4000/api/1/envelope`
(loopback-хоста нет вообще). Заголовок `X-Forwarded-Proto: https` —
как обращается прокси Coolify (иначе `force_ssl` даёт 301).

| Замер | Результат |
|---|---|
| c=64, 40 000 запросов | **~17 900 rps**, latency avg 3.58 мс, 100% `200`, 0 ошибок |
| Дренаж пайплайна | все 40 000 событий → issue (`times_seen=40000`); XPENDING группы 177 → **0** за секунды после нагрузки |

**Вывод:** честный потолок приёма одной ноды — **~18 000 событий/с**,
на порядок с лишним выше «единиц тысяч событий/с» из ADR-0004; пайплайн
разгребает буфер быстрее, чем идёт приём. Windows-замер выше был
целиком артефактом loopback (17 900 против 715 на том же железе). Redis
Streams (спайк ADR-0005: 150k+ XADD/s) — не узкое место.

Дальнейшее (при реальной нагрузке): профилировать батч-вставки в
ClickHouse и upsert issues в PG — именно они, а не HTTP-приём, станут
границей на устойчивом потоке.
